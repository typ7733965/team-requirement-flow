# 任务

通过 docs/<需求名>/ 文档 + git 历史，恢复跨会话/跨人的需求上下文，让用户与 Claude 都"对齐到当前进度"。

> **核心理念**：设计方案 + 实施步骤拆分表 + git commit message 是上下文的载体。
> 不依赖会话历史，不依赖记忆，**新会话也能在 30 秒内进入状态**。

## 1. 解析参数

- **需求名**：必填。对应 `docs/<需求名>/` 目录。
- 如果用户没传需求名：先 `ls docs/` 列目录，再用 AskUserQuestion 让用户选。

## 2. 加载需求全部文档

按顺序：

1. **Glob** `docs/<需求名>/*.md` 列出所有文件
2. **Read** 每个文件（不要 limit/offset，要看全）。重点章节：
   - 设计方案：`需求与目标` / `整体架构` / `关键设计决策` / `实施步骤拆分` / `风险`
   - 前端文档：`接入顺序` / `Checklist`
3. 如果存在多个版本（v1/v2 等），用最新版

## 3. 加载 git 历史

```bash
git log --oneline -100
git log --all --oneline -100 --grep="<需求名关键词>"
git status
git diff --stat HEAD
```

目的：
- 通过 commit message 关键词识别哪些"实施步骤拆分"已完成
- 通过 working tree 状态识别有无未提交的进行中改动
- 通过 diff stat 快速看本地未 commit 的范围

## 4. 与"实施步骤拆分"对账

将设计方案中的「实施步骤拆分」每一行与 git log 对照：

- commit message 关键词命中 → 该步骤已完成，记录 commit hash
- 工作区有相关文件改动但无 commit → 该步骤进行中
- 既无 commit 也无未提交改动 → 该步骤未开始

## 5. 输出"需求恢复报告"

按以下结构输出（**不要先 Read 设计方案就总结，必须读完再写**）：

```markdown
# 需求恢复报告：<需求名>

## 1. 需求一句话摘要
（1-2 句，从设计方案"需求与目标"提炼）

## 2. 关键设计决策回顾
（列出设计方案"关键设计决策"中的 3-5 条，每条一行）

## 3. 涉及仓库与模块
（列出涉及的代码仓库，每个仓库标注角色：proto / 业务 / 网关 等）

## 4. 实施进度

| # | 步骤 | 状态 | 证据 |
|---|------|------|------|
| 1 | xxx | ✅ 已完成 | commit abc1234 |
| 2 | xxx | ✅ 已完成 | commit def5678 |
| 3 | xxx | 🟡 进行中 | 工作区有未提交改动：xxx 文件 |
| 4 | xxx | ⏳ 未开始 | - |

## 5. 进行中事项细节
（如有 🟡：列出未提交改动的文件清单，简述改动方向）

## 6. 下一步建议
（基于"实施步骤拆分"的下一行 + 进行中事项的收尾，给出 2-3 条具体动作）

## 7. 需要用户确认的事项
（如发现设计方案有 TODO 占位、或 git 历史与设计方案有偏离，列出来让用户拍板）

## 8. 文档索引
- 设计方案：`docs/<需求名>/xxx.md`
- 前端文档：`docs/<需求名>/xxx.md`（如有）
- 其他：xxx
```

## 6. 输出后的动作

- 询问用户：「以上恢复对吗？要继续推进第 N 步吗？」
- 如果用户说"继续"：建议调用 `/team-requirement-flow:implement-from-doc <需求名> step=N`
- 如果用户说"先看看 xxx 文件"：用 Read 打开
- 如果用户说"方案要调整"：建议先 Edit 设计方案再继续实施

## 7. 关键原则

- **不要凭印象**：不要根据需求名"猜"做了什么；必须 Read + git log 对账
- **不要伪造进度**：找不到 commit 就老实写"未开始"，不要为了"看起来已完成"而瞎标 ✅
- **不要重复整段方案**：恢复报告是摘要，不是设计方案的副本；冗长会让用户失去焦点
- **跨仓库需求**：需要分别 `cd` 到每个仓库跑 git log（或用 `git -C <repo> log`）
- **如果文档与代码漂移严重**：在第 7 节明确标出，让用户决定是修文档还是改代码

## 8. 严禁

- **不要直接开始改代码**：本 Skill 只负责恢复上下文，**不写代码**。要写代码用 implement-from-doc。
- **不要省略 git log 步骤**：没有 git 证据的"已完成"标记是不可信的。
- **不要在缺失需求名时静默瞎选**：必须 AskUserQuestion 让用户确认。
