# 任务

为已设计/实施完成的需求生成「前端接入步骤」文档，输出到与设计方案同一目录。

## 1. 解析参数

- **需求名**：必填。对应 `docs/<需求名>/` 目录。
- **语言**：可选 `--lang zh`（默认）或 `--lang en`。决定文件名：
  - `zh`：`docs/<需求名>/前端接入<功能名>步骤.md`
  - `en`：`docs/<需求名>/frontend-integration.md`

## 2. 收集所有前端必需信息

按以下顺序读取（**全部必须**）：

1. **设计方案**：Glob `docs/<需求名>/*.md`，Read 全部
2. **接口定义**：根据项目类型 Grep 接口源码
   - Go gRPC：`proto/**/*.proto`、`internal/service/**/*.go`
   - REST：`router/`、`controller/`、`handler/`
   - GraphQL：`schema.graphql` 或 `*.gql`
3. **错误码源码**：Grep `enum/`、`errors/`、`constants/` 找到错误码字符串与触发条件
4. **网关路由**：如有网关层（gateway / BFF），Grep 路由注册代码确认 URL 与中间件
5. **配置项**：项目内是否有需要前端默认填写的字段（如本次的 `country_code = +86`）

## 3. 与用户对齐前端接入边界

用 AskUserQuestion 询问（按需）：
- 前端对接形式：H5 / SDK / iframe / 小程序？
- 是否涉及鉴权（Token / Cookie / Session）？
- 是否涉及"配置端"操作（运营在 B 端配置后 C 端才能用）？
- 错误码是否需要给具体文案建议？还是只给"是否可重试"标签？
- 是否有前端必须默认带的硬编码值（这条很关键，避免联调踩坑）？

## 4. 生成「前端接入步骤」文档

按以下结构生成（**章节必须齐全，无内容则写"本期无"**）：

## 5. 关键质量要求

- **错误码字符串必须与源码一致**：每条错误码必须用 Grep 确认字面量值，不要从设计方案抄（设计方案可能与实现漂移）
- **success code 数值必须确认**：很多框架用 0、200、20000 不一，**必须 Grep `SuccessCode` / `OK` 等常量**确认值
- **流程图用 ASCII 而不是 Mermaid**：兼容所有 Markdown 渲染器，包括 GitLab/GitHub 之外的内部 Wiki
- **不写废话**：每个章节都要有可操作内容；缺信息就写"本期无"
- **硬编码约定要重点标注**：用 ★ 必读 / 高亮区块，避免前端漏看（如本次 `country_code = +86`）

## 6. 写完后的动作

1. Write 文件
2. 用户输出：
   - 文件绝对路径
   - 文档大纲
   - 提示"如果实现端有变化，可重新运行本 Skill 增量更新"
   - 提示"对接前可让 doc-reviewer subagent 审一遍"

## 7. 严禁

- **不要凭印象写错误码**：必须 Grep
- **不要复用过时模板**：每次都基于当前 docs/<需求名>/ 内容生成
- **不要把后端实现细节倒灌给前端**：前端只关心接口契约，不关心 Redis Key 长什么样
- **不要遗漏"硬编码约定"章节**：这是前端接入翻车的最常见点
