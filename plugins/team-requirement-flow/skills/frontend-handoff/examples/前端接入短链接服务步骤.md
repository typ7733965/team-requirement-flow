# 前端接入短链接服务步骤

> 本文档为 `frontend-handoff` Skill 的**虚构示例**。所有接口路径、错误码、success code 均为示意。

## 一、总体顺序与全景流程图

```
阶段 A：创建短链接
  用户输入 long_url ─► POST /sl/create ─► 拿到 short_code ─► 展示短链接

阶段 B：访问短链接
  用户点击短链接 ─► 浏览器访问 /sl/{short_code} ─► 服务端 302 ─► 落到 long_url

阶段 C：查看统计（可选）
  仪表盘列表 ─► GET /sl/stat/{short_code} ─► 展示 PV / UV
```

接入顺序：**A → B → C**，A、B 是必接，C 可选。

---

## 二、阶段 A：创建短链接

### 2.1 接口

`POST /sl/create`

请求体：

```json
{
  "long_url": "https://example.com/very/long/path?utm=xxx",
  "custom_code": "myname",
  "ttl_days": 30
}
```

- `long_url` 必填，必须以 `http://` 或 `https://` 开头
- `custom_code` 可选，4～16 位字母数字
- `ttl_days` 可选，默认 30，范围 1～365

响应：

```json
{
  "code": 0,
  "data": {
    "short_code": "abcd1234",
    "expire_at": 1736000000000
  }
}
```

`expire_at` 单位为**毫秒**。

### 2.2 前端处理

- 拿到 `short_code` 后拼接 BASE_URL：`{BASE_URL}/sl/{short_code}`
- 复制按钮 + 二维码生成是前端职责

---

## 三、阶段 B：访问短链接

短链接是浏览器直接访问的，**前端不需要调用任何接口**。

服务端会返回 `302` + `Location: <long_url>`，浏览器自动跳转。

边界情况：

- 如果短码不存在 → 服务端返回 `404` + JSON `{ code: "SL_NOT_FOUND", ... }`
- 如果短码已过期 → 返回 `410` + JSON `{ code: "SL_EXPIRED", ... }`

如果产品方希望"过期/不存在"也走前端友好提示页，可由网关做 302 到前端 `/short-error?reason=expired`。

---

## 四、阶段 C：查看统计

### 4.1 接口

`GET /sl/stat/:short_code`

响应：

```json
{
  "code": 0,
  "data": {
    "short_code": "abcd1234",
    "pv": 12345,
    "uv": 678,
    "last_visit_at": 1735999000000
  }
}
```

### 4.2 前端处理

仪表盘列表中 `pv` / `uv` 直接展示数字；`last_visit_at` 用本地时区格式化。

---

## 五、接口速查表

| 阶段 | Method | URL | 鉴权 | 调用时机 |
|------|--------|-----|------|---------|
| A | POST | `/sl/create` | 需登录 | 用户提交"生成短链接"表单 |
| B | GET | `/sl/:short_code` | 不需要 | 浏览器自动访问 |
| C | GET | `/sl/stat/:short_code` | 需登录 | 仪表盘加载 |

## 六、错误码与前端处理对照表

> ⚠️ 字面量已 Grep 源码确认（`enum/shortlink_error.go`）。前端按字符串匹配，**不要按数字**。

| 错误码字面量 | 触发接口 | 前端建议处理 |
|------------|---------|-------------|
| `SL_INVALID_URL` | A | 表单红字提示"请输入合法 URL" |
| `SL_CODE_TAKEN` | A | 表单红字提示"短码已被占用" |
| `SL_TTL_OUT_OF_RANGE` | A | 表单红字提示"有效期 1~365 天" |
| `SL_RATE_LIMIT` | A | Toast 提示"创建过频，稍后再试" |
| `SL_NOT_FOUND` | B / C | B 由网关跳错误页；C 列表展示"未找到" |
| `SL_EXPIRED` | B / C | B 由网关跳过期页；C 列表展示"已过期" |

success code 是 `0`（不是 200，也不是 20000）。前端 `if (resp.code !== 0) { ... }` 判错。

## 七、硬编码约定（★ 必读）

以下约定散落在多个接口里，前端必须默认带：

1. **请求头 `X-App-Id`**：固定为 `web-shortlink`，所有接口必须带
2. **`ttl_days` 缺省值**：前端不传时按 `30` 处理（与服务端默认值一致）
3. **`short_code` 大小写敏感**：服务端按字面量匹配，不要做 `toLowerCase()`
4. **BASE_URL**：测试环境 `https://sl-dev.example.com`，生产环境 `https://sl.example.com`，由 `process.env.SHORTLINK_BASE_URL` 注入

## 八、联调 FAQ

### Q1：POST /sl/create 总是返回 `SL_INVALID_URL`

排查：
- 是否带了 `http://` / `https://` 前缀？
- 是否做了 `encodeURIComponent`？该接口要的是**原始 URL**，不要 encode

### Q2：访问短链接返回 200 而不是 302

可能原因：
- 浏览器开了 `Disable redirects` 调试选项
- 网关层未透传 302（向后端确认 `proxy_redirect off`）

### Q3：本地开发跨域

请求 `/sl/create` 时配置 vite/webpack 的 dev proxy，把 `/sl` 转发到 `https://sl-dev.example.com`。
