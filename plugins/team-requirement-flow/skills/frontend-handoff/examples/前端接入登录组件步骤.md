## 1. 总体顺序与全景流程图

### 1.1 接入顺序（按此推进，先做完一阶段再下一阶段）

```
阶段 A：B 端配置端落地     →   阶段 B：C 端登录链路落地    →   阶段 C：个人中心落地
（运营平台拖组件 + 配置）        （send_code → login）         （whoami / update / logout）
```

> 强烈建议：**先把阶段 A 跑通**（拿到一个已发布、已含登录组件的应用 uuid），再启动阶段 B；阶段 B 跑通后再做阶段 C。三个阶段互相不阻塞，但跨阶段联调容易踩坑。

### 1.2 C 端登录全景时序图


## 2. 阶段 A：B 端运营平台配置（运营操作 / 前端协助预演）

> 这部分**前端不写代码**，但**必须知道运营操作了什么**——因为 C 端的行为完全由这里的配置决定。

### 2.1 操作步骤（运营平台 B 端）

| 步骤 | 操作 | 备注 |
|-----|------|------|
| 1 | 进入低代码编辑器 → 拖入 `auth_login` 组件 | 一个应用的所有页面**只允许出现一个** auth_login 组件 |
| 2 | 配置 `providers`（登录方式） | 一期仅勾选 `phone` 一项；email/wechat/qq 暂不可用 |
| 3 | 配置 `token_ttl_days`（Token 有效期） | 仅可选 1 / 7 / 14 / 30 / 90 天 |
| 4 | 配置 `token_refresh_strategy`（续期策略） | `sliding`：whoami/refresh 自动续期；`fixed`：到期必须重登 |
| 5 | 配置 `multi_device_policy`（多端策略） | 三选一：`allow`（默认，允许多端在线，单用户最多 10 个 token）；`single`（**严格单端**，已有在线设备时新登录会被拒绝并返回 `DEVICE_CONFLICT`）；`kick_old`（**新登录踢旧端**，旧设备 token 立即作废） |
| 6 | 配置 `login_lock_enabled`（登录锁定） | true：连续 5 次失败锁 15 分钟；false：不锁 |
| 7 | 配置 `config_codes.phone`（短信模版码） | 必填，对应消息中心已配置的短信模版 |
| 8 | 保存并**发布应用** | 后端在发布回调中按需建表 + 写 cfg 缓存 |

### 2.2 发布后后端做了什么（前端理解上下文）

1. 扫描所有页面 → 检测到 `auth_login` 组件
2. 若该应用首次含登录组件：建一张 `user_app_users_xxx` 表存用户数据
3. 把组件配置写入 Redis：`game_operation:user_auth:cfg:{app_key}:{app_uuid}`
4. C 端登录链路从此 cfg 缓存取配置（永不回源）

### 2.3 前端验证方法

发布完成后，前端可以通过任何已登录的内部账号调一次 inner-gateway 的 `/game_operation/user_auth/send_code`（带正确的 app_uuid），如果返回 200 + cooldown_sec=60，说明阶段 A 配置已落地，可以进入阶段 B。

> **若返回错误码** `APP_NOT_SUPPORT_LOGIN`：说明应用未含 auth_login 组件或未发布，回到 §2.1 检查。

## 3. 阶段 B：C 端登录链路接入

### 3.1 前置准备（一次性）

```ts
// 全局常量（建议放在 env / config）
const API_BASE = 'https://<open-gateway-domain>/game_operation/user_auth'
const DEFAULT_COUNTRY_CODE = '+86'   // ★ 必读 §8
const TOKEN_KEY = 'user_auth_token'
const EXPIRE_KEY = 'user_auth_expire_at'
const DEVICE_ID_KEY = 'user_auth_device_id'
```

### 3.2 进入 H5 页面时的初始化（必做）

```
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 1：localStorage 取 device_id                                 │
│   - 取不到 → 生成 UUID v4，写入 localStorage（详见 §7）          │
│   - 取到 → 直接用                                                │
├─────────────────────────────────────────────────────────────────┤
│ 步骤 2：localStorage 取 token                                     │
│   - 取不到 → 跳到 §3.3 走"未登录态"渲染                          │
│   - 取到 + expire_at < now → 视为过期，清空，走"未登录态"          │
│   - 取到 + 未过期 → 调 whoami 进一步校验                          │
├─────────────────────────────────────────────────────────────────┤
│ 步骤 3：whoami 校验                                              │
│   - 200 + 用户信息 → 已登录态，渲染业务页                         │
│   - 错误码 TOKEN_INVALID → 清空 token，走"未登录态"               │
│   - 网络错误 → 重试一次；仍失败提示"网络异常"                    │
└─────────────────────────────────────────────────────────────────┘
```

### 3.3 未登录用户：发送验证码

**调用接口**：`POST /game_operation/user_auth/send_code`

```ts
{
  "provider": "phone",
  "country_code": "+86",        // ★ 必传，§8
  "identifier": "13800000000",  // 不含国家码
  "app_uuid": "<当前应用的 uuid>"
}
```

**前端时机**：
- 用户填完手机号、点"获取验证码"按钮时调
- 发送成功后**立即用 `cooldown_sec` 启动倒计时**禁用按钮，倒计时未结束不允许重发

**响应**：
```ts
{
  "cooldown_sec": 60,   // 60s 内不可重发
  "ttl_sec": 300        // 验证码 5 分钟内有效
}
```

**异常处理见 §6**。

### 3.4 未登录用户：提交登录

**调用接口**：`POST /game_operation/user_auth/login`

```ts
{
  "provider": "phone",
  "country_code": "+86",        // ★ 必传
  "identifier": "13800000000",
  "code": "123456",             // 6 位验证码
  "app_uuid": "<应用 uuid>",
  "device_id": "<localStorage 中的 device_id>"  // ★ 必传，§7
}
```

**前端时机**：
- 用户输入完验证码、点"登录"按钮时调
- **不要**在用户每次按键时预校验，等用户主动点确认

**成功响应**：
```ts
{
  "token": "abc...32 字符",
  "expire_at": 1761720000000,    // 毫秒时间戳
  "user": {
    "uuid": "...", "username": "...", "nickname": "...",
    "phone": "...", "country_code": "+86", "avatar": "...",
    "gender": 0, "register_at": ..., "last_login_at": ...
  }
}
```

**前端处理**：
1. 把 `token`、`expire_at` 存 localStorage
2. 把 `user` 信息更新到全局状态（vuex/pinia/redux）
3. 后续所有需要登录态的请求都把 `Authorization: Bearer <token>` 带上

### 3.5 已登录态：携带 token 调用接口

**所有需要登录态的接口**（whoami / refresh / logout / update_profile）都需带：

```
Authorization: Bearer <token>
```

> 如果你用 axios，建议在 `request interceptor` 统一注入。

### 3.6 Token 过期前的续期

| 续期策略 | 前端做什么 |
|---------|------------|
| `sliding`（滑动） | **无需主动调 refresh**，只要持续调 whoami（或任何受保护接口），后端会自动续期；`expire_at` 会从 whoami 响应中读出更新 |
| `fixed`（固定） | 不可续期，到期必须重新登录；前端检测到 `expire_at` 临近过期直接引导用户重登 |

> 一期默认建议运营选 sliding。

**`POST /refresh`** 仅在 sliding 策略下作为"用户主动续期按钮"使用（一般不会暴露给终端用户）。fixed 策略调用会返回错误码 `STRATEGY_NOT_ALLOW`。

### 3.7 注销

**调用接口**：`POST /game_operation/user_auth/logout`

```ts
{
  "scope": "current"   // current=仅当前 token；all=该用户所有活跃 token
}
```

**前端时机**：
- 用户点"退出登录"
- whoami 返回 `TOKEN_INVALID`（虽然已经无效，调一次保证清理；后端会幂等处理）

**前端处理**：
1. 不论后端是否成功，**立即清空** localStorage 的 token / expire_at
2. **保留** device_id（同一设备下次登录复用）
3. 跳到登录页 / 切回未登录态

## 4. 阶段 C：个人中心接入

### 4.1 进入个人中心页

调一次 `whoami` 拉最新用户信息（不要直接用 login 时存的，可能已经被改过）。

```
GET 用户信息：whoami → 渲染昵称/手机号/头像/性别/注册时间
```

### 4.2 修改个人资料

**调用接口**：`POST /game_operation/user_auth/update_profile`

```ts
{
  "nickname": "新昵称",        // 非空字符串则更新；空字符串=不更新
  "avatar": "https://...",     // 非空字符串则更新
  "gender": 1                  // 0/1/2 中之一则更新；-1=不更新
}
```

**前端时机**：用户点"保存"按钮时调，不要每个字段单独调（合并提交）。

**响应**：
```ts
{
  "user": { /* 更新后的完整 UserInfo */ }
}
```

**前端处理**：
1. 用响应里的 user 整体替换全局状态里的用户信息（避免本地拼接出错）
2. 个人中心页面用响应数据重新渲染

### 4.3 注销

复用 §3.7 的接口与流程。

## 5. 接口速查表

| # | URL（path） | 方法 | 鉴权 | 调用时机 |
|---|------|------|------|----------|
| 1 | `/game_operation/user_auth/send_code` | POST | 平台签名 | 用户点"获取验证码"按钮 |
| 2 | `/game_operation/user_auth/login` | POST | 平台签名 | 用户输入完验证码点"登录" |
| 3 | `/game_operation/user_auth/whoami` | POST | 平台签名 + Bearer Token | 进入需登录页面时校验登录态、个人中心拉信息 |
| 4 | `/game_operation/user_auth/refresh` | POST | 平台签名 + Bearer Token | （仅 sliding）用户主动续期按钮，几乎不用 |
| 5 | `/game_operation/user_auth/logout` | POST | 平台签名 + Bearer Token | 用户点"退出登录" |
| 6 | `/game_operation/user_auth/update_profile` | POST | 平台签名 + Bearer Token | 个人中心点"保存"按钮 |

> **平台签名**：open-gateway 自带的 sign 校验，前端 SDK 已封装。本登录组件不要求前端额外做事。
>
> **Bearer Token**：`Authorization: Bearer <token>` 形式，由前端在 §3.5 注入。

### 5.1 通用响应结构

所有接口外层包 `common_resp`，前端按以下结构解构：

```ts
{
  "code": 20000,                  // 20000=成功；非 20000=失败
  "msg": "OK" | "<错误码字符串>",  // 失败时为错误码（详见 §6）
  "data": { /* 业务字段 */ }       // 成功时业务数据
}
```

**判断成功的标准统一为 `code === 20000`**（后端 `toolEnum.SuccessCode = 20000`）。失败时直接看 `msg` 字段做错误分支处理。

> 常见非成功码参考：`40000` 业务失败、`50000` 内部错、`50001` 参数错、`401` HTTP 状态码（VerifyUserToken 中间件 token 缺失，详见 §6.1）。具体业务失败必须看 `msg` 字符串。

## 6. 错误码与前端处理对照表（★ 必读）

> 错误码定义在 `internal/enum/user_auth.go:59-74`。前端按 `msg` 字段字符串匹配。

| 错误码 | 含义 | 触发接口 | 前端建议处理 |
|--------|------|----------|--------------|
| `APP_NOT_PUBLISHED` | 应用未发布 | send_code, login | 提示"应用尚未发布"；联运营。**不可重试** |
| `APP_NOT_SUPPORT_LOGIN` | 应用未含登录组件 | send_code, login | 提示"应用未启用登录"；联运营。**不可重试** |
| `PROVIDER_NOT_SUPPORT` | 登录方式不支持 | send_code, login | 提示"该登录方式暂不可用"；切到其他方式。**不可重试** |
| `CONFIG_CODE_MISSING` | 短信模版未配置 | send_code | 提示"短信模版未配置，请联系运营"。**不可重试** |
| `TOO_FREQUENT` | 发送过于频繁（60s 冷却内重发） | send_code | 沿用本地倒计时；服务端兜底已拦下，提示"请稍候再试" |
| `CODE_INVALID` | 验证码错/过期/未发送 | login | 提示"验证码错误或已过期"；引导用户重新获取 |
| `LOCKED` | 账号锁定（连续 5 次失败 / 15 分钟） | login | 提示"登录失败次数过多，请 15 分钟后再试"；可禁用登录按钮 |
| `DEVICE_CONFLICT` | 设备冲突（multi_device_policy=`single` 且新 device_id 与已有在线设备不同） | login | 仅 `single` 策略会触发；提示"该账号已在其他设备登录，请先在其他设备退出再登录"。前端**不要重试同一登录请求**——必须等其他端 logout 后才能换设备登录。注意：同一 device_id 复登（同设备同浏览器 localStorage 未清）后端会自动复用 token 不会冲突 |
| `DEVICE_ID_INVALID` | device_id 格式错 | login | 提示"设备异常，请刷新重试"；清掉本地 device_id 重新生成（§7）后重试 |
| `TOKEN_INVALID` | Token 失效（未传/已过期/被踢） | whoami, refresh, logout, update_profile | **立即清空本地 token**，跳登录页 |
| `STRATEGY_NOT_ALLOW` | 策略不允许（fixed 策略下调 refresh） | refresh | 不应触发（前端按 §3.6 判断）；若触发则不要再尝试 refresh |
| `SEND_FAIL` | 短信发送失败（运营商错误） | send_code | 提示"验证码发送失败，请稍后重试"；可立即重试（冷却已回滚） |
| `USER_BANNED` | 用户被封禁 | login, whoami | 提示"账号已被封禁，请联系客服"。**不可重试** |
| `USER_GONE` | 用户已注销 | whoami, login | 提示"账号不存在或已注销"；清本地 token，回登录页 |

### 6.1 通用规则

1. **任何接口**返回 `code !== 20000`：先看 `msg` 是否在上表里命中，命中则按上表处理；未命中则提示 `msg` 原文（运营/网络异常）
2. **HTTP 5xx / 网络断开**：toast 提示"网络异常"，**不要重试登录类接口**（避免短信重发等副作用）
3. **HTTP 401**（VerifyUserToken 中间件返回）：与 `TOKEN_INVALID` 同处理——清 token 跳登录页

## 7. device_id 设备号生成与存储规范（★ 必读）

### 7.1 作用

- 后端用它做"多端策略"判断（参见 §2.1 步骤 5）：
  - `single` 模式：识别"是否同一设备复登"——同 device_id 自动复用并删旧 token；不同 device_id 拒绝并返回 `DEVICE_CONFLICT`
  - `kick_old` 模式：新登录踢旧端时按 device_id 识别需要踢哪些 token
  - `allow` 模式：device_id 仍记入 token payload，但不参与冲突判定
- 不同浏览器/不同设备视为不同 device_id
- 同一设备下次登录**复用同一个 device_id**，不要每次重新生成（重新生成会被 `single` 策略当作新设备拒绝）

### 7.2 生成规则（后端硬约束）

| 项 | 约束 | 说明 |
|----|------|------|
| 长度 | 1 - 128 字符 | 实际建议 36 位（标准 UUID v4） |
| 字符集 | `[A-Za-z0-9_-]` | **不允许**空格、中划线以外的标点、冒号、点等 |
| 唯一性 | 同一设备唯一 | 跨页面、跨刷新一致 |

> 不符合上述规则后端会返回 `DEVICE_ID_INVALID`。

### 7.3 推荐生成方式（任选其一）

#### 方式 A：标准 UUID v4（最推荐）

```ts
// 浏览器原生（HTTPS 环境可用）
const deviceId = crypto.randomUUID()
// 例：'a1b2c3d4-5e6f-7890-abcd-ef1234567890'  长度 36，符合字符集
```

#### 方式 B：手写 UUID v4 兼容老浏览器

```ts
function genDeviceId() {
  return 'xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx'.replace(/[xy]/g, c => {
    const r = Math.random() * 16 | 0
    const v = c === 'x' ? r : (r & 0x3 | 0x8)
    return v.toString(16)
  })
}
```

#### 方式 C：用第三方库（uuid / nanoid）

```ts
import { v4 as uuidv4 } from 'uuid'
const deviceId = uuidv4()
```

### 7.4 存储规则

```ts
const DEVICE_ID_KEY = 'user_auth_device_id'

function getDeviceId() {
  let id = localStorage.getItem(DEVICE_ID_KEY)
  if (!id) {
    id = crypto.randomUUID()
    localStorage.setItem(DEVICE_ID_KEY, id)
  }
  return id
}
```

### 7.5 何时不要清 device_id

| 场景 | 是否清 device_id |
|------|------------------|
| 用户注销（logout） | ❌ 保留（同设备下次登录复用） |
| Token 过期 | ❌ 保留 |
| 用户切换账号 | ❌ 保留（同设备同 device_id） |
| `DEVICE_ID_INVALID` 错误 | ✅ 清并重新生成（异常恢复） |
| 用户清浏览器缓存 | ✅ 自然丢失（下次进页面会自动重新生成） |

## 8. 国家码 country_code 必读（★ 必读）

### 8.1 当前产品形态

> **产品原型未设计国家码选择 UI**。

为了让后端接口正常工作，**前端在 send_code / login 请求里必须把 `country_code` 字段写死为 `"+86"`**：

```ts
{
  "provider": "phone",
  "country_code": "+86",   // ← 写死
  "identifier": "13800000000",
  ...
}
```

### 8.2 为什么不能省略

后端在拼接 Redis Key、查询用户、调消息中心时使用的是 `country_code + identifier`（即 `+8613800000000`），如果省略 `country_code`：
- send_code 会报参数错误（`手机登录 country_code 不能为空`）
- login 也会报同样错误
- 即使绕过校验，验证码 Key 拼出来也是 `+13800000000`，与发送时的 Key 不一致，验证码永远校验失败

### 8.3 二期扩展

二期产品如果增加国家码下拉选择，把硬编码的 `"+86"` 改成 UI 选项即可，**接口字段无需变更**。

### 8.4 字段格式

| 写法 | 是否合法 |
|------|---------|
| `"+86"` | ✅ 推荐 |
| `"86"` | ❌ 缺 `+` 后端不识别 |
| `"+86 "` | ❌ 含空格 |
| `""` 或不传 | ❌ 后端校验失败 |

## 9. 常见联调问题与排查

| 现象 | 可能原因 | 排查 |
|------|----------|------|
| send_code 收不到短信 | 短信模版未配置 / 发送失败 | 看错误码：`CONFIG_CODE_MISSING` 联运营；`SEND_FAIL` 让运营查消息中心日志 |
| login 永远 `CODE_INVALID` | country_code 不一致 / 验证码过期 | 1. 确认 send_code 和 login 传的 country_code 一致；2. 验证码 5 分钟内有效，超时重发 |
| 登录后 whoami 立刻 `TOKEN_INVALID` | Authorization Header 没带 / 格式错 | 浏览器开发者工具看请求头，必须是 `Authorization: Bearer <token>`，注意 `Bearer` 后面**有空格** |
| 登录成功但其他设备的 token 突然失效 | multi_device_policy=`kick_old`（新登录踢旧端） | 这是预期行为。旧设备下次调 whoami / 受保护接口会拿到 `TOKEN_INVALID`，按 §6 处理（清 token 跳登录页） |
| `DEVICE_ID_INVALID` | localStorage 里写脏值 | 清掉 `user_auth_device_id` 重新生成（用 §7.3 方式 A 最稳） |
| `APP_NOT_SUPPORT_LOGIN` | app_uuid 错 / 应用未发布 | 1. 检查 app_uuid 是否正确；2. 联运营确认应用已含登录组件并发布 |
| token 一段时间后突然失效 | sliding 策略下没有任何活跃请求 → 自然过期 | 这是预期行为，重新登录即可 |
| 跨域问题 | open-gateway 域名 CORS 配置 | 联后端运维加白名单 |
