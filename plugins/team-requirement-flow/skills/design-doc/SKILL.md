参数：$ARGUMENTS



# 任务

为用户输入的需求生成结构化的设计方案文档，并在 `docs/<需求名>/` 目录下沉淀。

## 1. 解析参数

从 `$ARGUMENTS` 中解析：
- **需求名**：必填。第一个非选项参数。允许中英文，但**不要包含空格**（用 `-` 或 `_` 替代）。
- **语言**：可选 `--lang zh`（默认）或 `--lang en`。决定文档语言与目录命名风格：
  - `zh`：目录 `docs/<需求名>/`，文件 `<需求名>设计方案_Claude.md`
  - `en`：目录 `docs/<requirement-name>/`，文件 `design.md`

如果用户未提供需求名，先用 AskUserQuestion 询问。

## 2. 探索项目上下文

按以下顺序读取（每项可选，不存在跳过即可）：
1. 项目根 `README.md`
2. 项目根 `CLAUDE.md` 与 `~/.claude/CLAUDE.md`
3. `docs/` 目录下已有的方案文档（用 Glob `docs/**/*.md`），了解项目历史决策与文档风格
4. 项目根的 `go.mod` / `package.json` / `Cargo.toml` / `requirements.txt` 等，识别技术栈
5. 一级目录结构（用 Glob `*/`），理解项目分层

## 3. 与用户对齐需求边界

**关键步骤，不要跳过。** 用 AskUserQuestion 询问以下事项（按需选择，不必每个都问）：

- **需求范围**：要解决什么问题？哪些不在本期范围？
- **关键决策点**：可能存在多种方案时（如同步/异步、新建表/扩展表、技术选型等）让用户选择
- **跨仓库**：是否涉及多个仓库的协同改动？
- **前端是否参与**：是否需要后续生成「前端接入步骤」文档？
- **预期接入方式**：是 RPC、HTTP API、消息队列、还是 SDK？

## 4. 创建目录与文档

```bash
mkdir -p docs/<需求名>/
```

按以下结构生成「设计方案」文档（中文版示例，英文版镜像翻译）：

```markdown
# <需求名>设计方案

> 作者：[用户名 or Claude 协作]
> 创建日期：YYYY-MM-DD
> 文档版本：v1.0

## 一、需求与目标

### 1.1 业务背景
（为什么要做、当前的痛点）

### 1.2 目标
（要达成的可度量结果）

### 1.3 范围
- ✅ 本期包含：...
- ❌ 不在范围：...
- 🔮 后续二期：...

## 二、整体架构

### 2.1 全局视图
（ASCII 时序图 / 模块依赖图）

### 2.2 涉及组件
| 组件/仓库 | 职责 | 是否新增 |
|----------|------|---------|

### 2.3 关键链路
（端到端调用顺序，标注每一步发生在哪个仓库/模块）

## 三、数据模型

### 3.1 数据库表
（如有）字段、索引、迁移策略

### 3.2 缓存设计
（如有）Redis Key 命名规范、TTL、多租户隔离策略

### 3.3 消息/事件
（如有）topic / event payload

## 四、接口设计

### 4.1 内部接口（RPC / 函数签名）
### 4.2 对外接口（HTTP / SDK）
请求/响应字段、字段含义、必填可选

## 五、错误码

| 错误码 | 含义 | 触发条件 | 调用方处理建议 |
|--------|------|----------|----------------|

## 六、关键设计决策

> 写"为什么这样做"，不写"做什么"。每条决策包含：候选方案 → 选择 → 理由

### 决策 N：xxx
- 候选 A：...
- 候选 B：...
- 选择：...
- 理由：...

## 七、实施步骤拆分

> 每一步对应一个独立 commit。粒度参考：能在 1-2 小时内完成、能独立测试。

| # | 步骤 | 涉及文件/模块 | 验证方式 |
|---|------|---------------|----------|
| 1 | ... | ... | ... |

## 八、风险与回滚

### 8.1 已识别风险
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|

### 8.2 回滚策略
（数据迁移 / 配置回滚 / 灰度回退）

## 九、附录

### 9.1 名词表
### 9.2 参考资料
```

## 5. 写入文件后的动作

1. 用 Write 工具写入 `docs/<需求名>/<文件名>.md`
2. 给用户输出：
   - 文件绝对路径
   - 文档大纲（前述九节标题）
   - 提示"如需修改/补充，可直接编辑该文件，或告诉我哪一节需要调整"
   - 提示"实施时使用 `/team-requirement-flow:implement-from-doc <需求名>`"
   - 如果用户表示需要前端接入，提示"前端文档使用 `/team-requirement-flow:frontend-handoff <需求名>`"

## 6. 重要原则

- **决策必有理由**：第六节是设计方案的灵魂，不能只罗列结论。
- **跨仓库需求要画清调用关系**：在第二节用 ASCII 图说明哪些仓库参与、调用方向。
- **错误码列全字符串值**：第五节的"错误码"列必须是字符串字面量（如 `"USER_GONE"`），不是数字。
- **不要凭空捏造**：如果某节信息缺失（如还没决定用哪个 Redis Key 前缀），写 `<!-- TODO: 待 xxx 同学补充 -->` 占位。
- **对齐项目历史**：步骤 2 读到的 `docs/` 已有文档是文档风格的标尺，新文档要与之对齐。
- **AskUserQuestion 不要超过 4 个**：一次问完最关键的，剩余可在写文档过程中边写边问或留 TODO。
