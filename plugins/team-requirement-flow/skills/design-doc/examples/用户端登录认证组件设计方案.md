## 一、背景与目标

### 1.1 背景

当前用户端低代码平台已具备页面搭建能力（7 个基础组件 + SSG 静态生成 + 事件流），但缺少面向 C 端用户的**身份认证能力**。游戏官网、活动页、H5 个人中心等场景需要用户登录后访问个性化内容。

### 1.2 目标

设计**两个内置组件**：

| 组件 code | 中文名 | 作用 |
|---------|------|-----|
| `auth_login` | 登录组件 | 嵌入页面的登录表单，完成手机号/邮箱验证码登录 |
| `user_profile` | 个人中心组件 | 头像入口 + 气泡卡片，展示登录态信息 + 跳转个人中心 + 退出登录 |

落地要求：

1. **一阶段只交付手机号验证码登录**，但配置结构按"多 Provider"建模，邮箱/微信/QQ 等后续接入零结构改动；
2. 用户数据**按 `user_apps.uuid` 应用级隔离**——每个应用一张独立用户表；
3. 短信通道**直接复用消息中心**（`grpc_clients.EventManagerClient().Send`）；
4. 登录鉴权**不复用组件事件流**，单开 `/user_auth/*` 独立 HTTP 路由，**两个网关均挂载**：`inner-gateway` 供内网自测、`open-gateway` 供 C 端调用；
5. 所有样式字段下沉为前端透传 `interface{}`，后端零解析。

## 二、整体架构

### 2.1 请求链路

```
C 端浏览器 (Nuxt SSG + CSR)
   │
   │  HTTPS  POST /user_auth/send_code / login / logout / refresh / whoami
   ▼
┌─────────────────────────┐     ┌─────────────────────────┐
│   open-gateway (外网)    │     │  inner-gateway (内网)    │
│   面向 C 端用户          │     │  内网联调 / 自测         │
└────────────┬────────────┘     └────────────┬────────────┘
             │   · 解析 AppKey（X-App-Key / 域名反查）
             │   · 解析 UserToken（Authorization: Bearer xxx，可选）
             │   · 中间件：VerifyUserToken（whoami/refresh/logout/业务接口需校验）
             │     校验维度：token 存在 + payload.app_key == ctx.AppKey + payload.app_uuid == ctx.AppUuid
             │     （多租户：项目=AppKey，项目下多应用=AppUuid）
             ▼
game-operation (gRPC)
   │   internal/service/user_auth.go       gRPC Handler
   │   internal/logic/user_auth.go         业务逻辑
   │   internal/logic/user_auth_table.go   动态用户表创建/查询
   │   internal/dao/user_auth.go           动态表 DAO
   │   utils/custom/user_page/component/
   │       auth_login.go                   登录组件 DSL
   │       user_profile.go                 个人中心组件 DSL
   ▼
依赖
   · Redis    验证码、Token 缓存、失败锁定计数
   · MySQL    user_apps（新增 user_table 字段）+ 动态用户表 user_app_users_{ts}
   · EventManager (RPC)  发送短信
```

> **网关分工**：两个网关使用同一份 Controller / 中间件代码，只是部署位置和入口不同。`inner-gateway` 走公司内网，便于开发联调；`open-gateway` 暴露给公网，配合域名/CDN 直连 C 端。

### 3.2 动态用户表

**表名生成规则**（参考 `internal/logic/custom_func.go:1322` 的 `GenerateCustomTableName`）：

```go
// internal/logic/user_auth_table.go
func GenerateUserTableName() string {
    return fmt.Sprintf("user_app_users_%s", utils.GenerateTimeBasedID())
}
```

> 命名对齐：`custom_func_xxx` → `user_app_users_xxx`。表名与应用 1:1 绑定，存入 `user_apps.user_table`。

**表结构**：

| 字段 | 类型 | 说明 |
|-----|------|-----|
| id | bigint pk auto_increment | 主键 |
| uuid | varchar(64) uniq | 用户 uuid（对外暴露的 user_id） |
| username | varchar(64) uniq | 用户名（首次登录自动生成，规则见 §6.2.2） |
| nickname | varchar(64) | 用户昵称（默认 = 脱敏手机号） |
| country_code | varchar(8) | 手机号国家码（如 `+86`），与 phone 联合唯一 |
| phone | varchar(32) | 手机号（不含国家码，如 `13800000000`） |
| email | varchar(128) NULL | **预留**邮箱字段，**允许 NULL 而非空串**，预先建唯一索引（MySQL 允许多个 NULL 不冲突） |
| avatar | varchar(512) | 头像 url |
| gender | tinyint | 性别 0 未知 / 1 男 / 2 女 |
| register_at | bigint | 注册时间（ms） |
| last_login_at | bigint | 最近登录时间（ms） |
| last_login_ip | varchar(64) | 最近登录 IP |
| status | tinyint | 1 正常 / 2 封禁 / 3 注销 |
| attrs | json | 扩展属性兜底字段（实名状态、昵称变更次数等，避免后续频繁加列） |
| created_at | bigint | |
| updated_at | bigint | |
| deleted_at | `*gorm.DeletedAt` | 软删（与 `user_apps` 类型保持一致） |

**索引**：
- `uniq_uuid (uuid)`
- `uniq_phone (country_code, phone)` ← **联合唯一**
- `uniq_username (username)`
- `uniq_email (email)` ← 预先建立，二期邮箱登录零改动
- `idx_last_login (last_login_at)`

### 3.3 Redis Key 规范

> **identifier 统一约定**：
> - `provider = phone` → `identifier = country_code + phone`（如 `+8613800000000`，**全局唯一形式**，避免不同国家相同号码冲突）；
> - `provider = email` → `identifier = email` 原样；
> - 其他 provider 后续接入时，按"完整可对外标识"统一拼接。
>
> **多租户 key 隔离规则**（重要）：
> - 平台支持**跨项目导入导出应用**功能 → 不同项目（AppKey）下可能存在 `app_uuid` 相同的应用；
> - **所有按应用维度的 key 必须以 `{app_key}:{app_uuid}` 双层前缀隔离**，仅以 `{app_uuid}` 隔离会发生跨项目串台；
> - **唯一例外**：`user_auth:token:{token}` 不加项目前缀——token 是 32 字符随机串全局唯一，且其 payload 内已记录 `app_key + app_uuid`，VerifyUserToken 中间件会比对（§9）。

| Key | Value | TTL | 用途 |
|-----|-------|-----|------|
| `user_auth:code:{app_key}:{app_uuid}:{provider}:{identifier}` | 6 位验证码 | 5 min | 验证码内容 |
| `user_auth:cd:{app_key}:{app_uuid}:{provider}:{identifier}` | `1` | 60 s | 发送冷却（1 分钟内不可重复发送）|
| `user_auth:lock:{app_key}:{app_uuid}:{provider}:{identifier}` | 失败次数 | 15 min | 登录失败锁定计数（按 provider 维度独立计数）|
| `user_auth:token:{token}` | `{app_key, app_uuid, user_uuid, provider, device_id, ttl_days, expire_at, ip, ua}` | 配置决定 | 会话（Opaque Token，全局唯一，不加项目前缀）|
| `user_auth:user_tokens:{app_key}:{app_uuid}:{user_uuid}` | SortedSet（token→expire_at） | 最长 90d | 用户→活跃 token 列表（支持单端踢出） |
| `user_auth:cfg:{app_key}:{app_uuid}` | JSON：`{providers, token_ttl_days, token_refresh_strategy, multi_device_policy, login_lock_enabled, config_codes}` | 永久（无 TTL）| 发布回调时主动写入；登录链路直接读取，避免穿透到 MySQL 解析 page_conf |
| `user_auth:user:{app_key}:{app_uuid}:{user_uuid}` | JSON：用户表行（脱敏后） | 5 min | whoami / 业务接口加速；登录、改资料时主动失效（`DEL`） |
| `user_auth:risk:{app_key}:{app_uuid}` | Hash：`{ip_check_enabled: bool, ...}` | 永久 | 应用级风控开关（如 IP 首段校验，§10.2） |

> `user_auth:cfg:{app_key}:{app_uuid}` 由 `EnsureUserTable` 之后紧跟一步 `WarmAuthConfigCache` 写入。组件配置变更只通过"重新发布"才会生效，**不**在编辑保存时刷新——保持发布即生效的语义一致。

## 四、登录组件（`auth_login`）

### 4.1 组件配置

原型呈现为**单一组件**，所有配置集中在右侧属性面板。

```go
// utils/custom/user_page/component/auth_login.go
package component

const CodeAuthLogin = "auth_login"

// LoginProviderKind 登录方式（一期仅实现 phone，结构按多 provider 扩展）
const (
    LoginProviderPhone  = "phone"  // 手机号验证码（一期实现）
    LoginProviderEmail  = "email"  // 邮箱验证码（预留）
    LoginProviderWeChat = "wechat" // 微信（预留）
    LoginProviderQQ     = "qq"     // QQ（预留）
)

// TokenRefreshStrategy Token 刷新策略
const (
    TokenRefreshSliding = "sliding" // 滑动过期（默认）：活跃即续期
    TokenRefreshFixed   = "fixed"   // 固定过期：到期即失效
)

// MultiDevicePolicy 多端登录策略
const (
    MultiDeviceAllow         = "allow"          // 允许多端（默认）
    MultiDeviceSingle        = "single"         // 仅单端在线（新登录失败）
    MultiDeviceKickOld       = "kick_old"       // 新登录踢出旧端
)

// AuthLogin 登录组件
type AuthLogin struct {
    *UserComponentBase

    // 1. 启用的登录方式（多选；一期仅允许包含 phone，其他值前端禁用并提示"敬请期待"）
    //    存储为切片是为了未来直接扩展，后端校验"包含未实现的 provider 时拒绝发布"
    Providers []string `json:"providers"`

    // 2. 登录成功跳转的目标页面 uuid（前端落地）
    RedirectPageUuid string `json:"redirect_page_uuid"`

    // 3. Token 有效期（单位：天） 可选：1 / 7(默认) / 14 / 30 / 90
    TokenTtlDays int32 `json:"token_ttl_days"`

    // 4. Token 刷新策略  sliding(默认) / fixed
    TokenRefreshStrategy string `json:"token_refresh_strategy"`

    // 5. 多端登录策略  allow(默认) / single / kick_old
    MultiDevicePolicy string `json:"multi_device_policy"`

    // 6. 登录失败锁定（开关）
    // 开启后连续 5 次失败 → 锁定 15 分钟（按 provider 维度独立计数）
    LoginLockEnabled bool `json:"login_lock_enabled"`

    // 7. 消息中心推送模版（按 provider 分别配置）
    //    示例: { "phone": "sms_login_code", "email": "email_login_code" }
    //    一期前端只需填 phone；二期接入邮箱时补 email key 即可，后端零改动
    ConfigCodes map[string]string `json:"config_codes"`

    // 8. 其他前端样式配置（interface{}，后端不解析）
    WebConfig interface{} `json:"web_config"`
}
```

> **重要约束**：以上配置**仅影响新签发的 Token**。已签发并写入 Redis 的 Token，其 TTL、所属策略均不会因组件配置变更而追溯调整——避免一次配置改动导致所有在线用户被踢出。

### 4.2 默认值约定

| 字段 | 默认值 |
|-----|-------|
| Providers | `["phone"]` |
| TokenTtlDays | `7` |
| TokenRefreshStrategy | `sliding` |
| MultiDevicePolicy | `allow` |
| LoginLockEnabled | `false` |

### 4.3 原型对应的 UI 元素

登录组件前端渲染逻辑（概念）：

```
┌──────────────────────────────┐
│ [按 Providers 渲染顶部切换]   │ ← len(Providers)==1 时不渲染切换条
├──────────────────────────────┤
│  [+86 ▾] [ 手机号输入框     ]│ ← provider=phone 时
│  [ 验证码 ] [发送验证码 60s ]│ ← 倒计时 / 锁定提示
│  [ □ 我已阅读并同意《协议》] │
│  [    登 录    ]             │
│  错误提示区域（锁定/错误）   │
└──────────────────────────────┘
```

> 一期 `Providers` 固定为 `["phone"]`，前端属性面板的"登录方式"为多选 Checkbox，但 `email/wechat/qq` 选项 disabled 并提示"敬请期待"。前端自行实现倒计时、锁定提示、错误展示等；后端只返回状态码与剩余时间。

## 五、个人中心组件（`user_profile`）

### 5.1 需求背景

以**头像图标**嵌入页面，点击弹出**气泡卡片**，展示当前登录用户信息、跳转"个人中心"页、退出登录。

### 5.2 配置

按用户要求，**仅需要一个前端透传 `interface{}`**，后端零字段解析。

```go
// utils/custom/user_page/component/user_profile.go
package component

const CodeUserProfile = "user_profile"

// UserProfile 个人中心组件
// 后端仅做组件类型识别，所有展示/交互行为（默认头像、尺寸、未登录时的显示、气泡卡片内容与跳转）
// 全部由前端基于 WebConfig + 登录态自行渲染。
type UserProfile struct {
    *UserComponentBase

    // 唯一配置：前端样式/行为透传
    // 建议前端约定包含：
    //   - defaultAvatar       未登录或未设置头像时的占位图
    //   - avatarSize          头像尺寸（px）
    //   - unloggedDisplay     未登录时展示样式（隐藏 / 显示登录按钮 / 显示默认头像）
    //   - profilePageUuid     点击"个人中心"跳转的页面 uuid
    //   - popupPlacement      气泡位置（top/bottom/left/right）
    //   - customCSS           自定义样式
    WebConfig interface{} `json:"web_config"`

```

### 5.3 前端渲染约定

- **未登录态**：根据 `web_config.unloggedDisplay` 渲染（隐藏 / 登录按钮 / 默认头像点击打开登录弹窗）；
- **已登录态**：从 `/user_auth/whoami` 拉取用户信息，渲染头像；点击弹出气泡卡片展示 `nickname / username / phone(脱敏)`，提供"个人中心"与"退出登录"按钮；
- **退出登录**调用 `POST /user_auth/logout`。

## 六、HTTP 接口定义（独立路由 `/user_auth/*`）

### 6.1 路由注册（双网关）

`inner-gateway`（内网联调）与 `open-gateway`（C 端调用）使用同一份路由代码，仅初始化文件分别注册：

```go
// inner-gateway/index/route/game_operation_router.go
// open-gateway/index/route/game_operation_router.go  （内容相同）
g := r.Group("/user_auth",
    middleware.VerifySign(),    // 校验 AppKey 签名
    middleware.AppContext(),    // 注入 app_key / app_uuid
)
{
    g.POST("/send_code", ctrl.SendCode)
    g.POST("/login",     ctrl.Login)
    g.POST("/logout",    middleware.VerifyUserToken(), ctrl.Logout)
    g.POST("/refresh",   middleware.VerifyUserToken(), ctrl.Refresh)
    g.POST("/whoami",    middleware.VerifyUserToken(), ctrl.Whoami)
}
```

> `VerifyUserToken` 中间件读取 `Authorization: Bearer <token>`，从 Redis 取出会话并**同时校验 `payload.app_key == ctx.AppKey` 与 `payload.app_uuid == ctx.AppUuid`**（多租户：项目=AppKey，应用=AppUuid，二者均匹配才可放行），写入 `ctx.user_uuid`。`open-gateway` 与 `inner-gateway` 各自维护一份相同实现（也可抽到公共库）。

### 6.2 接口列表

#### 6.2.1 `POST /user_auth/send_code` 发送验证码

**Request**
```json
{
  "provider":     "phone",            // phone / email
  "country_code": "+86",              // provider=phone 时必填；provider=email 时忽略
  "identifier":   "13800000000",      // 手机号(不含国家码) 或 邮箱
  "app_uuid":     "xxxxxxxx"          // 应用 uuid（也可由 Header 注入）
}
```

> 后端落库与缓存使用的 **完整 identifier** 由后端拼接：`provider=phone` 时为 `country_code + identifier`（即 `+8613800000000`）；`provider=email` 时直接使用 `identifier`。下文 Redis Key 中的 `{identifier}` 即指此完整形式。

**Response**
```json
{
  "code": 0,
  "data": { "cooldown_sec": 60, "ttl_sec": 300 }
}
```

**后端逻辑**
1. 通过 `app_uuid` 查 `user_apps`：
   - 若 `status != 1（已发布）` → 返回 `400 APP_NOT_PUBLISHED`（开发预览态不允许调真实发码）；
   - 若 `user_table` 为空 → 返回 `400 APP_NOT_SUPPORT_LOGIN`；
2. 读 `user_auth:cfg:{app_key}:{app_uuid}`（B1：发布回调写入），缺失则回源解析 page_conf 一次并补写；校验 `provider ∈ Providers`（一期仅 `phone`），否则 `400 PROVIDER_NOT_SUPPORT`；
3. 拼接 `fullIdentifier`（见上）；生成 6 位数字验证码 `code`；
4. **原子写冷却 key**：`SET user_auth:cd:{app_key}:{app_uuid}:{provider}:{fullIdentifier} 1 NX EX 60`
   - 若返回 `0`（已存在）→ 返回 `429 TOO_FREQUENT`；
   - 若返回 `1`（成功占位）→ 继续；
5. 写验证码：`SET user_auth:code:{app_key}:{app_uuid}:{provider}:{fullIdentifier} {code} EX 300`；
6. 从 cfg 缓存取 `ConfigCodes[provider]`：
   - 不存在则回滚冷却 key（DEL）+ 返回 `400 CONFIG_CODE_MISSING`；
7. 调用消息中心发送（见 §7）；若发送失败，**DEL 冷却 key**（让用户立即重试）+ 返回 `500 SEND_FAIL`。

#### 6.2.2 `POST /user_auth/login` 登录

**Request**
```json
{
  "provider":     "phone",
  "country_code": "+86",                 // provider=phone 时必填；email 时忽略
  "identifier":   "13800000000",
  "code":         "123456",
  "app_uuid":     "xxxxxxxx",
  "device_id":    "browser-fingerprint"  // 必填，多端策略与 token 复用依赖该字段，格式见 §6.5
}
```

**Response**
```json
{
  "code": 0,
  "data": {
    "token":      "opaque_xxxx",
    "expire_at":  1735689600000,
    "user": {
      "uuid":          "u-xxx",
      "username":      "u_8a2f1b3c",
      "nickname":      "138****0000",
      "country_code":  "+86",
      "phone":         "13800000000",
      "avatar":        "",
      "gender":        0,
      "register_at":   1735000000000,
      "last_login_at": 1735689500000
    }
  }
}
```

**后端逻辑**
1. 查 `user_apps.user_table` 得用户表名；空 → `400 APP_NOT_SUPPORT_LOGIN`；
2. 读 `user_auth:cfg:{app_key}:{app_uuid}`，校验 `provider ∈ Providers`，否则 `400 PROVIDER_NOT_SUPPORT`；校验 `device_id` 格式（见 §6.5），不合法 → `400 DEVICE_ID_INVALID`；
3. 拼接 `fullIdentifier`（同 §6.2.1）；
4. 校验锁定：若 `LoginLockEnabled=true` 且 `GET user_auth:lock:{app_key}:{app_uuid}:{provider}:{fullIdentifier}` ≥ 5 → `423 LOCKED`；
5. 校验验证码：`GET user_auth:code:{app_key}:{app_uuid}:{provider}:{fullIdentifier}` 不匹配
   - `INCR user_auth:lock:{app_key}:{app_uuid}:{provider}:{fullIdentifier}` 并 `EXPIRE 900`（15 min）；
   - 返回 `400 CODE_INVALID`；
6. 匹配则 `DEL` 验证码 key、`DEL` 锁定计数；
7. 按 `(country_code, phone)` 或 `email` 查用户：
   - 未注册 → 自动创建（生成 `uuid` 用 `id.GetWithUUID()`、`username = "u_" + uuid 后 8 位`，写入失败重试 3 次再生成新后缀）；
   - 注册时 `nickname` 默认按 §10 脱敏算法生成；
8. 更新 `last_login_at` / `last_login_ip`，并 `DEL user_auth:user:{app_key}:{app_uuid}:{user_uuid}` 失效缓存；
9. **多端策略处理**（`user_auth:user_tokens:{app_key}:{app_uuid}:{user_uuid}` 为 ZSET，member=token，score=expire_at）：

   ```
   newToken  = GenerateRandomString(32)   // 32 字符 urlsafe，~192 bit 熵
   ttlMs     = TokenTtlDays * 86400000
   expireAt  = now + ttlMs
   payload   = {
     app_key, app_uuid, user_uuid, provider, device_id,
     ttl_days  = TokenTtlDays,        // ★ A5：冻结到 token，sliding 续期不读最新组件配置
     refresh   = TokenRefreshStrategy, // ★ 一并冻结，避免运营改了策略影响在线 token
     expire_at, ip, ua, login_at
   }

   // ★ 三分支统一前置（B1）：清理 ZSET 中已过期的 token 项
   //   token key 因 EX 自然过期会消失，但 ZSET 不会自动清——必须主动 GC：
   //     - allow 分支不清理会 ZCARD 偏大、过早踢掉合法旧 token；
   //     - single 分支不清理会误判 ZCARD>0 拒绝合法新登录；
   //     - kick_old 分支不清理会徒增对已过期 token 的无效 DEL。
   ZREMRANGEBYSCORE user_auth:user_tokens:{app_key}:{app_uuid}:{user_uuid} 0 {now}
   //   语义：score=expire_at <= now 视为过期，整体清掉

   switch MultiDevicePolicy:

     allow:
       1) 写新 token：SET user_auth:token:{newToken} payload EX ttl
       2) ZADD user_tokens:* expireAt newToken
       3) 若 ZCARD > 10（上限） → ZRANGE 取最旧若干，逐一 DEL token key 并 ZREM

     single:
       1) 检查同 device_id 是否已有活跃 token：
          遍历 ZSET 中每个 token，GET payload，比对 device_id
          - 命中 → 复用：覆盖旧 token（先删旧 token key 与 ZSET 项），再走 allow 写入流程
          - 未命中且 ZCARD > 0 → 返回 409 DEVICE_CONFLICT（任何其他设备在线）
          - 未命中且 ZCARD = 0 → 走 allow 写入流程

     kick_old:
       1) 写新 token（先写后踢，避免空窗）：
          SET user_auth:token:{newToken} payload EX ttl
          ZADD user_tokens:* expireAt newToken
       2) 遍历 ZSET 中除 newToken 外的所有旧 token：
          DEL user_auth:token:{oldToken}
          ZREM user_tokens:* oldToken
       3) 旧端下次请求 whoami/refresh/logout → 401 → 前端自行清登录态
   ```

10. 返回 `token + expire_at + user`。

**关键常量**
- 多端策略 `allow` 下单用户 token 上限：**10**（超出按 `score=expire_at` 升序踢最旧）；
- `username` 生成：`u_` + `uuid` 后 8 位（hex），唯一约束冲突时重试最多 3 次（每次重新生成 uuid 并取后 8 位）。

#### 6.2.3 `POST /user_auth/whoami` 获取当前用户

**Request**：Header `Authorization: Bearer <token>`

**Response**：与 `login.data.user` 相同结构。

**逻辑**：
- 读 `user_auth:token:{token}` 得 `payload`；
- **B2 用户缓存**：先读 `user_auth:user:{app_key}:{app_uuid}:{user_uuid}`，命中则直接返回；
  - 未命中 → 查应用用户表 → `SET user_auth:user:{app_key}:{app_uuid}:{user_uuid} {json} EX 300`；
- 若 `payload.refresh = sliding`：
  - **续期 TTL 使用 `payload.ttl_days`（冻结值）**，而非读取最新组件配置：
    - `EXPIRE user_auth:token:{token} payload.ttl_days * 86400`
    - `ZADD user_auth:user_tokens:{app_key}:{app_uuid}:{user_uuid} (now + payload.ttl_days*86400000) {token}`（更新 score）；
- 若 `payload.refresh = fixed`：直接返回，不续期。

#### 6.2.4 `POST /user_auth/refresh` 续期

**Request**：Header `Authorization: Bearer <token>`

仅在 `payload.refresh = sliding` 时可用。`fixed` 模式返回 `400 STRATEGY_NOT_ALLOW`。续期 TTL 同样取 `payload.ttl_days`（A5 冻结值），不读取最新组件配置。

#### 6.2.5 `POST /user_auth/logout` 注销

**Request**：Header `Authorization: Bearer <token>`

**Body**
```json
{ "scope": "current" }   // current(默认) / all
```

逻辑：
- `current` → `DEL user_auth:token:{token}` + 从 ZSET `user_auth:user_tokens:{app_key}:{app_uuid}:{user_uuid}` 移除（ZREM）+ `DEL user_auth:user:{app_key}:{app_uuid}:{user_uuid}` 失效用户缓存（B2：避免 5 分钟内同浏览器再登录拿到陈旧 last_login_at）；
- `all` 顺序：
  1. `ZRANGE user_auth:user_tokens:{app_key}:{app_uuid}:{user_uuid} 0 -1` 取出全部 token；
  2. 对每个 token 执行 `DEL user_auth:token:{token}`；
  3. `DEL user_auth:user_tokens:{app_key}:{app_uuid}:{user_uuid}`（整体清掉 ZSET，避免逐项 ZREM）；
  4. `DEL user_auth:user:{app_key}:{app_uuid}:{user_uuid}` 失效用户缓存。

### 6.4 登录态失效约定

> 当 Token 因过期、被踢出（kick_old）、单端冲突、或主动注销而失效时，**后端不会主动推送给前端**。

**约定流程**：
1. 前端发起任意需鉴权请求（如 `whoami` / 业务接口）；
2. `VerifyUserToken` 中间件读取 Redis 失败 → 返回 `401 TOKEN_INVALID`；
3. 前端拦截 401 → 清除本地 Token + 重置登录态全局 store → 视场景决定跳转登录页或刷新当前页；
4. **不引入 WebSocket / 长轮询** 来主动通知；C 端基于"懒感知"策略。

### 6.5 `device_id` 生成与校验约定

#### 背景

低代码平台一期面向 **PC 网页 + 移动 H5**，浏览器并没有"硬件设备 ID"。多端策略（`single` / `kick_old`）与"同设备 token 复用"必须依赖一个**前端可持久化、后端可识别**的标识符。

#### 前端生成规则（约定）

按以下顺序选取，**最终落到 `localStorage`**：

```js
// pseudo-code (前端实现，非后端代码)
const KEY = 'auth_device_id'
let deviceId = localStorage.getItem(KEY)
if (!deviceId) {
  if (window.crypto?.randomUUID) {
    deviceId = crypto.randomUUID()                  // ① UUID v4，36 字符
  } else {
    deviceId = 'dev_' + [...crypto.getRandomValues(new Uint8Array(16))]
                          .map(b => b.toString(16).padStart(2, '0')).join('')
  }
  localStorage.setItem(KEY, deviceId)
}
```

> **隐私模式 / 清除缓存的语义**：等价于"设备重置"——浏览器清缓存 = 卸载重装 App。`single` 模式下用户清缓存重新登录，会被视作新设备，触发 `409 DEVICE_CONFLICT`（须在另一端先 logout）；`kick_old` 模式下旧端会被踢。**该取舍前端登录页须有提示文案**。
>
> **跨域共享**：若同一应用挂多个二级域名，前端可考虑写到主域 cookie 同步；不同应用之间 device_id 互相独立（localStorage 天然隔离）。

#### 后端校验规则（B3）

```
长度：1 ≤ len(device_id) ≤ 128
字符集：[A-Za-z0-9_-]（含 UUID 标准字符）
```

不合法直接 `400 DEVICE_ID_INVALID`，不写 Redis、不消耗验证码（不消耗的前提：device_id 校验在验证码校验之前）。

#### App 端预留

未来上线原生客户端时，可按平台改为：
- iOS：`identifierForVendor`
- Android：`Settings.Secure.ANDROID_ID` 或自建 GUID
- 二者均落 keychain / SP，重装清空

接入只需替换前端获取代码，后端格式校验保持不变。

// internal/logic/user_auth.go
func (l *UserAuthLogic) sendSmsCode(

    }

}

## 八、应用用户表的动态创建时机

### 8.1 触发条件（按产品原型）

> **每次发布**应用收到构建成功回调时都调用 `EnsureUserTable`，并保证幂等：
> - 若 `user_apps.user_table` 已有值 → 跳过建表；
> - 若为空但页面含登录组件 → 创建；
> - 若为空且页面无登录组件 → 不创建。
>
> 这种"每次都查、空才建"的策略既覆盖"首次发布即含登录组件"也覆盖"先发不含登录、后期补加登录"的场景。
>
> **B1 cfg 缓存预热**：紧随 `EnsureUserTable` 之后调用 `WarmAuthConfigCache(appKey, appUuid)`，将该应用 `auth_login` 组件的关键配置（providers / token_ttl_days / token_refresh_strategy / multi_device_policy / login_lock_enabled / config_codes）整体写入 `user_auth:cfg:{app_key}:{app_uuid}`（无 TTL，每次发布回调覆盖）。登录链路读 cfg 缓存，避免穿透到 MySQL 解析 `page_conf`。
>
> 同一应用多个页面如同时含 `auth_login`（极端场景），以扫描到的第一个为准；产品上建议同应用仅一份登录组件配置。

### 8.2 具体流程

接入点：`internal/logic/user_app.go:498` 的 `UserAppLogic.BuildCallback`，**仅在 `req.BuildStatus == 1`（构建成功）分支** 调用 `EnsureUserTable` 与 `WarmAuthConfigCache`；构建失败分支保持现有逻辑（清理失败版本快照、置 status=2 下架），不建表也不写 cfg 缓存。

> **失败/异常容忍**：`EnsureUserTable` 或 `WarmAuthConfigCache` 内部错误不应阻断 `BuildCallback` 主流程（应用 status 已切换到 1），错误以 `logger.Warn` 记录后吞掉，等待下次发布回调自愈（`EnsureUserTable` 因 user_table 已空仍会重试建表；cfg 缓存缺失时 `send_code` 会回源解析 page_conf 并补写，§6.2.1 step 2）。

> **task_id 匹配**：现有 `BuildCallback` 已对 `req.TaskId` 与 `user_app_versions` 做匹配校验，防止旧任务回调覆盖新任务状态——天然保证 `EnsureUserTable` 不会被过期回调触发误建表。

### 8.3 建表代码骨架

```go
// internal/logic/user_auth_table.go
package logic

import (
    "context"
    "fmt"

    jsoniter "github.com/json-iterator/go"
    "gitlab.outer.staruniongame.com/centerserver/game-operation/global"
    "gitlab.outer.staruniongame.com/centerserver/game-operation/internal/dao"

// EnsureUserTable 应用发布回调中调用，幂等：user_apps.user_table 有值则直接返回
func EnsureUserTable(ctx context.Context, appKey, appUuid string) (tableName string, err error) {
    app, err := dao.NewUserAppDao(ctx).GetByUuid(appUuid)
    if err != nil {
        return "", err
    }
    if app.UserTable != "" {
        return app.UserTable, nil
    }
    hasLogin, err := HasLoginComponent(ctx, appUuid)
    if err != nil || !hasLogin {
        return "", err
    }

    tableName = fmt.Sprintf("user_app_users_%s", utils.GenerateTimeBasedID())

    db := global.GetFacade().GetAppStorage().GetDB(appKey)
    if err = db.Table(tableName).AutoMigrate(&model.UserAppUser{}); err != nil {
        return "", err
    }
    if err = dao.NewUserAppDao(ctx).UpdateUserTable(appUuid, tableName); err != nil {
        return "", err
    }
    return tableName, nil
}

// HasLoginComponent 扫描应用所有页面 page_conf，使用 jsoniter 严格解析，递归查找 auth_login 组件
func HasLoginComponent(ctx context.Context, appUuid string) (bool, error) {
    pages, err := dao.NewUserAppPageDao(ctx).ListByAppUuid(appUuid)
    if err != nil {
        return false, err
    }
    for _, p := range pages {
        if p.PageConf == "" {
            continue
        }
        var root map[string]interface{}
        if err := jsoniter.UnmarshalFromString(p.PageConf, &root); err != nil {
            continue // 单页解析失败不阻断，但应记日志
        }
        if findComponentByCode(root, component.CodeAuthLogin) {
            return true, nil
        }
    }
    return false, nil
}

// findComponentByCode 递归遍历 map / slice，匹配 "code" 字段等于 target 的对象
func findComponentByCode(node interface{}, target string) bool {
    switch v := node.(type) {
    case map[string]interface{}:
        if code, ok := v["code"].(string); ok && code == target {
            return true
        }
        for _, child := range v {
            if findComponentByCode(child, target) {
                return true
            }
        }
    case []interface{}:
        for _, item := range v {
            if findComponentByCode(item, target) {
                return true
            }
        }
    }
    return false
}
```

## 九、按应用隔离的实现要点

| 隔离点 | 实现 |
|-------|------|
| 用户数据 | 每个应用独立动态表 `user_app_users_{ts}`（§8） |
| Redis key | 全部按 `user_auth:{type}:{app_key}:{app_uuid}:...` 双层前缀隔离（§3.3）；唯一例外 `user_auth:token:{token}` 全局唯一不加前缀 |
| Token 作用域 | 验证 Token 时**同时校验** `payload.app_key == ctx.AppKey` 与 `payload.app_uuid == ctx.AppUuid`，缺一不可。多租户语义：**项目 = AppKey**（商户），**应用 = AppUuid**（项目下的应用），跨应用或跨项目盗用 Token 都会被拦截 |
| 跨项目导入应用 | 因 `app_uuid` 在被导入到新项目时仍可保留原值 → Redis key 必须 `{app_key}:{app_uuid}` 双前缀；若使用 `user_auth:cfg:{app_uuid}` 这类老 key 会发生跨项目串台 |
| 签名中间件 | `middleware.VerifySign` 已按 AppKey 校验，天然隔离项目维度 |
| DAO 访问 | `dao.NewUserAppUserDao(ctx, db, tableName)` 工厂，tableName 由 `app.UserTable` 注入 |
| 用户缓存 | `user_auth:user:{app_key}:{app_uuid}:{user_uuid}`，5 min TTL；登录、改资料、logout（current/all）时主动 DEL（§3.3） |
| cfg 缓存 | `user_auth:cfg:{app_key}:{app_uuid}`，发布回调覆盖写入，永不过期（§3.3 / §8） |

## 十、安全与合规

| 风险 | 对策 |
|-----|------|
| 验证码爆破 | 60s 冷却 + 5 分钟 TTL + 开关控制的 5 次错误锁定 15 分钟（按 provider 维度独立计数） |
| 验证码轰炸 | 同 IP 级限流（可在网关层基于 IP + identifier 做二级限流） |
| Token 泄露 | Opaque Token 服务端可即时撤销；whoami 链路可选校验 IP 首段（C2，默认关闭） |
| 多端策略 | `single` / `kick_old` 精确满足产品原型 |
| 跨应用越权 | Token 绑定 `AppKey + AppUuid`，每次请求强校验（A6） |
| 设备伪造 | `device_id` 后端做长度+字符集校验（B3），不合法直接拒绝；多端策略下"同 device_id 复用 token" 防止重复登录刷计数 |
| 手机号隐私 | `nickname` 默认按下方算法脱敏生成（C1） |
| 审计 | 登录/登出/踢出操作均记录日志（含 IP / UA / device_id） |

### 10.1 昵称脱敏算法（C1）

注册时 `nickname` 默认值由后端生成（用户登录后可在个人中心修改）：

```go
// internal/logic/user_auth.go
// MaskNickname 根据 provider 与 identifier 生成脱敏昵称
//   phone: 保留前 3 位 + 后 4 位，中间 4 位以 `*` 替换；不足 11 位时退化为前 3 + 后 2
//          示例: "13800000000" → "138****0000"
//   email: 保留 @ 前 1 位 + 后缀 @域名；本地部分超过 1 位时其余以 `*` 补齐
//          示例: "alice@example.com" → "a****@example.com"
//   其他 provider 暂未实现，默认返回 "user_" + uuid 后 6 位
func MaskNickname(provider, identifier string) string {
    switch provider {
    case LoginProviderPhone:
        n := len(identifier)
        if n >= 11 {
            return identifier[:3] + "****" + identifier[n-4:]
        }
        if n >= 5 {
            return identifier[:3] + "****" + identifier[n-2:]
        }
        return "user_" + identifier
    case LoginProviderEmail:
        at := strings.Index(identifier, "@")
        if at <= 0 {
            return "user_" + identifier
        }
        return identifier[:1] + "****" + identifier[at:]
    default:
        // 未实现 provider 走 fallback，由调用方传 uuid 后 6 位
        return identifier
    }
}
```

> 注意：phone 传入 `identifier` 应是不含国家码的纯手机号（来自登录请求 `identifier` 字段），落库 phone 字段同样不含国家码；`country_code` 单独存储。

### 10.2 IP 首段校验（C2，默认关闭，按需开启）

> 默认**关闭**——移动端跨基站、家宽拨号都会换 IP，强校验会误伤。仅在"高敏感"应用（如金融、企业内部）由运营在管理端为单个应用开启。

实现要点：
- 登录时 `payload.ip` 写入登录 IP；
- whoami / refresh / 业务接口（VerifyUserToken 中间件）若开启该开关，按下面规则比较：
  - **IPv4**：取请求 IP 与 `payload.ip` 各自的"首段"（点分十进制第一段，如 `223.5.5.5` → `223`），不一致 → `401 TOKEN_INVALID`；
  - **IPv6**：直接放过不校验（B3：移动端 IPv6 由运营商分配，跨基站频繁变化，强校验误伤大）；
  - 若 `payload.ip` 与请求 IP 协议族不同（如登录时是 IPv4、当前请求是 IPv6 或反之），同样放过——视作合规切换；
- 401 触发 → 前端清登录态走重新登录流程；
- 该开关存放在应用级配置（非登录组件配置，避免在 cfg 缓存里混入）：`user_auth:risk:{app_key}:{app_uuid}` 的 Hash 字段 `ip_check_enabled`。一期可硬编码 `false`，预留管理端入口。
