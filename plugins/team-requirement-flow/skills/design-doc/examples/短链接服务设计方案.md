# 短链接服务设计方案

> 本文档为 `design-doc` Skill 的**虚构示例**。所有接口、错误码、Redis Key 命名均为示意，不对应任何真实代码库。

## 一、需求与目标

### 1.1 背景

业务方需要一个**短链接服务**，把任意 URL 压缩成 8 位短码，访问短码 302 跳转到长链接。

### 1.2 范围

- ✅ HTTP API：长链接换短链接、短码 302 跳转
- ✅ 短码自定义（用户可指定，如 `/sl/myname`）
- ✅ 访问统计（PV / UV）
- ✅ 短码失效时间（TTL 1～365 天）

### 1.3 不在范围

- ❌ 短码权限控制（v2 再做）
- ❌ 反爬虫与限流（接入网关层统一处理）
- ❌ A/B Test 跳转（需求方未提）

## 三、数据模型

### 3.1 MySQL 表 `short_link`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 自增主键 |
| short_code | VARCHAR(16) UNIQUE | 短码 |
| long_url | VARCHAR(2048) | 长链接 |
| creator_id | BIGINT | 创建者 |
| ttl_days | INT | 失效天数 |
| expire_at | BIGINT | 失效时间戳（毫秒） |
| created_at | BIGINT | 创建时间戳（毫秒） |

索引：`short_code`、`creator_id + created_at`

### 3.2 Redis Key

| Key | 类型 | TTL | 说明 |
|-----|------|-----|------|
| `sl:{short_code}` | String | 与 expire_at 同 | 短码 → 长链接缓存 |
| `sl_seq` | Counter | 永久 | 自增序列，用于生成短码 |
| `sl_stat:{short_code}:pv` | Counter | 30 天 | PV 计数 |

## 四、接口设计

### 4.1 HTTP

| Method | Path | 说明 |
|--------|------|------|
| POST | `/sl/create` | 创建短链接 |
| GET | `/sl/:short_code` | 跳转短链接（302） |
| GET | `/sl/stat/:short_code` | 查询访问统计 |

### 4.2 RPC

```
service ShortLinkService {
  rpc CreateShortLink(CreateShortLinkReq) returns (CreateShortLinkResp);
  rpc ResolveShort(ResolveShortReq) returns (ResolveShortResp);
  rpc GetStat(GetStatReq) returns (GetStatResp);
}
```

### 4.3 请求 / 响应字段

`CreateShortLinkReq`: `long_url` (必填) / `custom_code` (可选) / `ttl_days` (默认 30)
`CreateShortLinkResp`: `short_code` / `expire_at`

## 五、错误码

> 字面量必须与代码保持一致，前端按字符串匹配。

| 错误码（字面量） | 触发条件 | 前端建议处理 |
|----------------|----------|-------------|
| `SL_INVALID_URL` | long_url 非法或为空 | 提示"请输入合法 URL" |
| `SL_CODE_TAKEN` | 自定义短码已被使用 | 提示"短码已被占用，换一个" |
| `SL_NOT_FOUND` | 短码不存在 | 跳转 404 页 |
| `SL_EXPIRED` | 短码已过期 | 跳转"链接已过期"页 |
| `SL_TTL_OUT_OF_RANGE` | ttl_days 超出 1～365 | 提示"有效期 1~365 天" |
| `SL_RATE_LIMIT` | 单用户创建过频 | 提示"创建过频，稍后再试" |

success code = `0`（约定，源码以 `enum.SuccessCode` 为准）

## 六、关键设计决策

### 6.1 短码生成策略

- 候选 A：MD5(long_url) 取前 8 位 → 可能碰撞，且相同 URL 总是相同短码
- 候选 B：自增 ID + Base62 编码 → 单调递增，不碰撞 ✅
- 候选 C：UUID → 太长，不适合短链接

**选 B**。理由：碰撞 0、生成快、可枚举。代价：短码可被遍历。本期需求不涉及隐私，可接受。

### 6.2 长链接合法性校验

- 候选 A：服务端校验 + 黑名单
- 候选 B：仅做基础格式校验，黑名单交给上游网关 ✅

**选 B**。理由：黑名单是公司级公共能力，不应在短链接服务里维护。

### 6.3 缓存策略

- Redis 缓存全量短码（TTL 与 expire_at 一致）
- 命中率 > 99% 时几乎不打 MySQL
- Cache miss → 回源 MySQL → 回填 Redis

## 八、风险与回滚

| 风险 | 应对 |
|------|------|
| Redis 故障 → 回源 MySQL 击穿 | 加单飞（singleflight）防止穿透 |
| 短码冲突（Base62 碰撞） | 自增 ID 单调，不会碰撞 |
| 长链接被恶意刷量 | 网关限流 + 黑名单 |

回滚：本服务为新增，可整体下线，不影响存量功能。
