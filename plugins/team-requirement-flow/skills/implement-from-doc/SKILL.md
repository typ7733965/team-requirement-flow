# 任务

读取设计方案，按步骤推进实施。每一步必须：编码 → 编译/测试通过 → 用户确认 → 进入下一步。

## 1. 解析参数

- **需求名**：必填。第一个非选项参数，对应 `docs/<需求名>/` 目录。
- **step**：可选 `step=N`。指定从第几步开始。不传则从"实施步骤拆分"中的第一个未完成步骤开始。

## 2. 加载设计方案与进度

1. **Glob** `docs/<需求名>/*.md` 找到设计方案（优先匹配名字含"设计方案" / "design"的文件）。
2. **Read** 完整设计方案，重点关注「实施步骤拆分」「关键设计决策」「错误码」「数据模型」四节。
3. 跑 `git log --oneline -50` 与 `git status`，识别哪些步骤已完成（通过 commit message 关键词或文件改动判断）。
4. 输出"进度判定"给用户：

```
设计方案：docs/<需求名>/<文件名>.md
共 N 步实施步骤：
  ✅ 1. xxx (commit abc1234)
  ✅ 2. xxx (commit def5678)
  ▶ 3. xxx <- 当前
    4. xxx
    ...
```

## 3. 推进当前步骤

对于"当前步骤"：

1. **重读设计方案中该步骤的描述**，理解涉及文件、验证方式
2. **用 TodoWrite 把步骤拆细**为 3-7 个 sub-task
3. **逐个 sub-task 推进**，每个完成后标记 completed
4. **遇到分歧**：用 AskUserQuestion 询问用户
5. **遇到缺失上下文**：read/grep/glob 项目源码，不要凭空假设
6. **写代码时遵守的硬约束**：
   - **不读不改**：不要改没读过的文件，先 Read 再 Edit
   - **跟随项目惯例**：命名、错误处理、目录组织参考同模块已有代码
   - **不过度工程**：不加未要求的功能、不重构未指定的代码
   - **不编造 API**：调用第三方/项目内函数前用 Grep 确认签名

## 4. 步骤完成验证

每步完成后必须做的事：

1. **编译/构建**：运行项目对应的 build 命令（go build / npm run build / 等）
2. **单元测试（如该步骤涉及）**：运行测试
3. **总结输出给用户**：
   - 改动的文件清单
   - 关键决策（与设计方案是否一致；如有偏离，写明原因）
   - 编译/测试结果
4. **询问用户**："本步骤完成，是否进入下一步？" 等用户确认。

## 5. 用户确认后的动作

- 如果用户说"继续" / "下一步"：进入下一步骤，回到 §3。
- 如果用户说"先 commit"：用 git 提交（commit message 包含步骤号 + 简述，遵守项目历史 commit 风格）。**只有用户明确要求才 commit。**
- 如果用户说"暂停"：把当前进度同步到设计方案的"实施步骤拆分"表（Edit 添加 ✅ 标记或 commit hash），让下次 resume-from-doc 能识别。
- 如果用户调整方向：建议先 Edit 设计方案再继续实施，避免方案与代码漂移。

## 6. 实施过程中的硬规则

- **不要跳步骤**：除非用户明确要求"先做第 N 步"。设计方案的步骤顺序是经过依赖排序的。
- **不要批量做多步**：一次只推进一步，便于 review 与回滚。
- **不要修改设计方案中"关键设计决策"章节**：那是项目级共识。如果发现决策需调整，**先停下来问用户**。
- **错误码必须用设计方案中定义的字符串字面量**：禁止临时编新错误码字符串。
- **跨仓库改动**：在每个仓库分别推进，每个仓库改完单独编译验证，不要一次跨多个仓库改完才编译。

## 7. 输出原则

- 多用 TodoWrite 维护步骤内的 sub-task
- 关键文件改动后，把 `file_path:line_number` 反馈给用户便于跳转
- 编译错误的输出原样贴出，不要总结
- 不要 emoji（除非用户明确要求）
